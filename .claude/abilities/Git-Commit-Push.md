# Git代码提交推送能力

## 核心职责
- 严格按照 规范执行 Git 代码提交和推送操作
- 确保提交消息符合 Angular 规范，内容详细且准确记录变更情况
- 提供完整的提交报告，确保代码变更可追溯、可审计

## 工作流程

### 步骤1：获取项目名称
- 提示用户输入项目名称
- 如果项目名称已知，则跳过此步骤
- 验证项目路径是否存在且为有效的 Git 仓库

### 步骤2：进入项目目录并且编译项目
- 切换到指定的项目目录
- 如果项目目录已知，则跳过此步骤
- 确认当前工作目录正确

### 步骤3：确认分支信息
- 确认当前所在分支：执行 `git branch --show-current`
- 确认需要提交的目标分支
- 如果分支信息已知，则跳过此步骤
- 如需切换分支，先确认工作区状态是否干净

### 步骤4：分析变更情况
- 执行 `git status` 查看工作区状态
- 执行 `git diff` 分析本次变更的具体内容
- 执行 `git diff --cached` 查看已暂存的变更（如有）
- 详细记录变更的文件列表、变更类型（新增/修改/删除）
- 分析变更的核心内容和影响范围

### 步骤5：暂存变更文件
- 根据变更情况，执行 `git add` 命令暂存文件
- 支持以下暂存方式：
  - `git add <file>`：暂存指定文件
  - `git add .`：暂存所有变更（需用户确认）
  - `git add -p`：交互式暂存（适用于部分提交）
- 确认暂存内容与预期一致

### 步骤6：组织提交消息
- 严格按照 Angular 规范组织提交消息
- 提交消息格式：
  ```
  <type>(<scope>): <subject>
  
  <body>
  
  <footer>
  ```
- **Type 类型说明**：
  | 类型 | 说明 | 示例 |
  |------|------|------|
  | feat | 新功能 | feat(auth): 添加用户登录功能 |
  | fix | 修复 Bug | fix(api): 修复接口超时问题 |
  | docs | 文档变更 | docs(readme): 更新安装说明 |
  | style | 代码格式（不影响逻辑） | style: 统一代码缩进格式 |
  | refactor | 重构代码 | refactor(utils): 优化工具函数结构 |
  | perf | 性能优化 | perf(query): 优化数据库查询性能 |
  | test | 测试相关 | test(user): 添加用户模块单元测试 |
  | chore | 构建/工具变动 | chore(deps): 更新依赖版本 |
  | revert | 回滚提交 | revert: 回滚上一次提交 |
  | ci | CI/CD 相关 | ci: 更新 GitHub Actions 配置 |

- **Scope 范围说明**：
  - 可选字段，表示影响范围
  - 常见范围：api、ui、db、auth、config、utils 等
  - 多个范围用 `/` 分隔，如 `feat(api/ui): xxx`

- **Subject 主题说明**：
  - 简短描述，不超过 50 个字符
  - 使用祈使句，首字母小写，结尾不加句号
  - 清晰说明"做了什么"，而非"怎么做的"

- **Body 正文说明**：
  - **强制要求**：正文内容必须详细，至少 50-100 字
  - 详细描述变更的原因、内容和影响
  - 使用列表形式说明具体变更点
  - 示例：
    ```
    feat(auth): 添加用户登录功能
    
    本次提交新增用户登录模块，包含以下变更：
    - 新增 LoginService 服务类，处理登录逻辑
    - 新增 TokenValidator 工具类，验证 Token 有效性
    - 新增 LoginController 控制器，提供登录接口
    - 更新 UserMapper，新增查询用户信息方法
    
    影响范围：用户认证模块
    测试情况：已通过单元测试和集成测试
    ```

- **Footer 页脚说明**：
  - 可选字段
  - 关联 Issue：`Closes #123`
  - 破坏性变更：`BREAKING CHANGE: xxx`

### 步骤7：执行提交
- 使用 `git commit` 命令提交代码
- 提交命令格式：`git commit -m "提交消息"`
- 多行提交消息使用 `git commit` 打开编辑器或使用多个 `-m` 参数
- 确认提交成功，记录提交哈希值

### 步骤8：推送代码
- 执行 `git push` 推送代码到远程仓库
- 如果是新分支，执行 `git push -u origin <branch-name>`
- 处理可能的推送冲突：
  - 如果推送失败，执行 `git pull --rebase` 后重新推送
  - 如果存在冲突，提示用户手动解决后再推送

### 步骤9：验证推送结果
- 执行 `git log -1` 查看最新提交记录
- 执行 `git status` 确认工作区状态
- 执行 `git branch -vv` 确认分支跟踪状态
- 验证远程仓库是否同步成功

### 步骤10：展示提交报告
- 生成完整的 Git 提交报告
- 报告用户提交结果和后续注意事项

## 强制规则

### 工具使用
- **强制使用 Bash 工具执行所有 Git 命令**
- 禁止使用其他方式操作 Git

### 提交消息规范
- **强制使用 Angular 规范组织提交消息**
- **提交消息正文强制详细，至少需要 50-100 字**
- **强制记录变更情况**，包括变更文件、变更内容、影响范围
- 提交消息必须使用中文（团队约定）

### 执行顺序
- 必须按照工作流程顺序执行，不可跳过任何步骤
- 每个步骤执行完成后，确认结果再进行下一步
- 遇到错误必须停止并报告，不可强行继续

### 安全检查
- 提交前必须确认变更内容与用户预期一致
- 推送前必须确认分支正确
- 禁止强制推送（`git push -f`）除非用户明确要求

### 信息确认
- 项目名称、分支信息等关键信息必须确认
- 如果信息未知，必须主动询问用户
- 变更内容必须向用户展示并确认

## 使用场景

### 场景1：常规代码提交
```
用户：帮我提交代码
助手：执行完整的提交流程，包括确认项目、分支、分析变更、组织消息、提交、推送、输出报告
```

### 场景2：指定分支提交
```
用户：帮我把代码提交到 feature/login 分支
助手：确认当前分支，如需切换则先切换，然后执行提交流程
```

### 场景3：部分文件提交
```
用户：只提交 src/utils 目录下的修改
助手：只暂存指定目录的文件，执行提交流程
```

### 场景4：修复 Bug 提交
```
用户：提交修复登录 Bug 的代码
助手：使用 fix 类型组织提交消息，详细描述 Bug 原因和修复方案
```

### 场景5：新功能提交

```
用户：提交新开发的用户注册功能

```

## 异常处理

### 推送冲突
1. 执行 `git pull --rebase origin <branch>`
2. 如有冲突，提示用户手动解决
3. 解决后执行 `git rebase --continue`
4. 重新推送 `git push`

### 工作区不干净
1. 提示用户当前有未提交的变更
2. 询问用户是否需要暂存或提交这些变更
3. 根据用户选择执行相应操作

### 分支不存在
1. 提示用户目标分支不存在
2. 询问用户是否需要创建新分支
3. 确认后执行 `git checkout -b <branch-name>`

### 权限问题
1. 报告具体的权限错误信息
2. 提示用户检查 Git 配置和权限
3. 建议联系仓库管理员

## 注意事项
- 提交前务必确认变更内容正确
- 敏感信息（如密钥、密码）禁止提交
- 大规模变更建议分批提交
- 重要提交建议打 Tag 标记
- 推送前确保代码已通过本地测试
